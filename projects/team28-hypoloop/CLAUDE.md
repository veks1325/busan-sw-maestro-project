## 0. 프로젝트 개요

**Hypo Loop**는 사용자가 등록한 "가설"을 ML 실험으로 자동 검증하는 **Auto Research Agent** 플랫폼이다.
사용자가 가설을 등록하면 → 에이전트가 실험을 설계·실행하고 → 점수와 분석 결과를 보고서로 돌려준다.

기술 기반: LangGraph 기반 에이전트 + FastAPI 백엔드 + 파일/SQLite 데이터 계층.

---

## 1. 데이터 계층 (전체 공통)

```
u_id (사용자)
└── project_id (프로젝트)
    ├── project.db (프로젝트별 로컬 SQLite)
    └── 가설_id (가설)
        ├── u_id_가설_id.yml   ← 백엔드 생성 (가설 메타)
        └── exp_id (실험)
            ├── exp_id.yml      ← AI팀 생성 (실험 설계)
            ├── status.yml      ← 실험 상태/점수 (파일 관리, AI팀 갱신)
            └── 학습용 기능코드  ← AI팀 생성
```

식별자 계층: `u_id → project_id → 가설_id → exp_id`

## 3. 영역별 책임 경계 (중요)

작업이 자기 영역을 넘어가지 않도록 아래 경계를 지킨다.

### 프론트 🟡
- 화면/UI 전반 (접속 첫화면, 가설 등록, 실험 진행 표시, 완성 보고서)
- 백엔드 API 호출로 데이터 표시

### 백엔드 ✅
- 담당 O: 식별자 계층 관리, 프로젝트별 SQLite 생성·연결·경로 정규화, `u_id_가설_id.yml` 생성, ready 트리거, 보고서용 데이터 집계
- 담당 X: 실제 ML 학습, `exp_id.yml` 생성, 실험 설계 내용, 병렬 실행, eda/재실험 분기
- 상세: `backend/BACKEND_TASK.md`

### AI팀 (에이전트) 🟡
- `exp_id.yml` (실험 설계: 피처/하이퍼파라미터/모델/산식) 생성
- 트레인 코드 작성 및 학습 실행
- `parallel_count` 만큼의 실험 병렬 실행/스케줄링
- `status.yml`에 점수·상태·분석 텍스트 갱신
- eda / 재실험 분기 로직

## 4. 공통 규약

### 파일/YML
- 백엔드가 생성: `u_id_가설_id.yml`, `status.yml` 골격
- AI팀이 생성/갱신: `exp_id.yml`, `status.yml`의 점수·상태·분석
- 실험 상태/점수는 **DB 테이블이 아니라 `status.yml` 파일로 관리** ✅

### Git / 협업
- main 직접 push 대신 **feature 브랜치 + PR** ✅
- 커밋 컨벤션: `feat:`, `fix:`, `chore:` 등
- 브랜치 보호·머지 전략은 팀 합의 사항 🟡

### 코드 스타일
- 함수에 타입 힌트 + docstring
- 영역 경계를 넘는 변경은 코드로 밀어붙이지 말고 **팀 질문으로 남길 것**

## 5. 확정 사항 (Decisions)

## 6. 결정 대기 (Open Questions)

팀에서 확정 후 이 문서와 해당 영역 문서에 반영한다.

- ⏳ **트리거 방식**: yml `ready` 폴링 vs 직접 API 호출 vs 메시지 큐
  - 병렬을 AI팀이 맡으므로, 데모 우선이면 폴링 / 병렬 본격화면 메시지 큐가 유력
- ⏳ **`status.yml` 동시 쓰기 충돌 처리**: 파일 락 vs 실험별 파일 분리(단일 writer 보장)
- ⏳ **프로젝트별 DB/파일 저장 경로 루트**: `data/projects/{project_id}/` 고정 vs 환경변수(`HYPOLOOP_DATA_ROOT`)
  - 추천 절충안: 환경변수로 받되 기본값 `data/projects/`
- ⏳ 프론트 / AI팀 영역 상세 문서 작성
